# /i18n

Audit a file or module for internationalisation compliance, and scaffold string registration when building new `_lib/` components.

Two tiers of work exist:

- **Tier 1 — V1 requirements (blocking).** No hardcoded user-visible strings in `_lib/`. Every string comes from `t()`. Strings registered via `defineStrings()` in `app/strings.js`. If `app/locales/` exists, all locale files must have key parity with `en.js`.
- **Tier 2 — Intl wrappers (deferred to V4).** `Intl.DateTimeFormat` / `Intl.NumberFormat` wrappers so dates and numbers respond to locale switches. Non-blocking until V4.

**Note:** The V1 locale system is built and active. `core/strings.js` exports `setLocale()`, `getLocale()`, and `defineStrings(obj, locale='en')`. Locale is persisted in `localStorage`. Language switching triggers `location.reload()`. Locale packs live at `app/locales/<locale>.js`.

---

## Usage

```
/i18n <filepath>           audit a file for hardcoded strings and t() compliance
/i18n locales              audit app/locales/ — check all locale packs have matching keys
```

---

## What counts as a user-visible string

Strings that must go through `t()`:
- Button labels, link text, headings, status messages
- Placeholder text, error messages, confirmation prompts
- Any text a user reads or acts on

Strings that do NOT need `t()`:
- HTML tag names, attribute names, CSS class names
- `customElements.define()` tag names
- IDB object store names, Store action types, event names
- Developer-facing console errors (these are internal, not UI)
- Log entries, test descriptions

---

## Tier 1 audit — `_lib/` files (blocking)

### 1. Scan for hardcoded strings in templates

Read every `template()` method and any string used in a DOM update method. Flag any string literal that is user-visible text — a word or phrase a user reads.

```js
// ❌ Hardcoded — fails
template() {
  return `<button>Reload</button>`;
}

// ✅ Correct — passes
template() {
  return `<button>${t('update.reload')}</button>`;
}
```

### 2. Check imports

Every `_lib/` file that calls `t()` must import from `_lib/core/strings.js`. Flag any file that uses `t()` without this import.

```js
import { t } from '../strings.js'; // ✅ correct relative path — t() only, not defineStrings
```

### 3. Check default string registration

`_lib/` components must NOT call `defineStrings()` themselves — that would bake English strings into the library layer. English defaults are registered by the app in `app/strings.js`, which must be the first import in `app/main.js`.

The `_lib/` component's only responsibility is to call `t('namespace.key')`. Flag if:
- A `_lib/` component calls `defineStrings()` (wrong layer — move to `app/strings.js`)
- A `_lib/` component uses `t()` without importing it from `strings.js`
- Any key used in a `t()` call has no corresponding entry in `app/strings.js` (will silently fall back to the key name)

### 4. Check `app/strings.js`

Verify that `app/strings.js` exists and is the first import in `app/main.js`. Check that every key used by `t()` in any imported `_lib/` component is registered. A mismatched or missing key silently falls through to the key name — flag any gap found.

---

## Tier 1 audit — `app/` components

`app/` components do not have the same restriction. They may use hardcoded strings if the app is single-language. However, if `app/locales/` exists, flag any hardcoded string that should be in the locale file.

---

## Locale key parity audit (`/i18n locales`)

Run whenever `app/locales/` exists with at least two locale files.

### Key parity check

Every key in every locale file must exist in `en.js` (the base locale). Every key in `en.js` must exist in every other locale file. Report:

```
Missing in fr.js: goal.add, goal.complete
Extra in fr.js (not in en.js): goal.archive   ← likely a renamed key
```

### Intl usage check (Tier 2 — advisory)

If the app uses raw `.toLocaleDateString()`, `.toLocaleString()`, `.toLocaleTimeString()`, or `new Intl.*` directly, flag them as advisory. When the V4 Intl wrapper module exists, these must go through it so they respond to locale changes.

## Scaffolding a new `_lib/` component with strings

When building a new `_lib/` component that needs user-visible text:

1. Choose a short, unique namespace (e.g. `'update-banner'`, `'toast'`)
2. List every user-visible string the component will show
3. Use `t('namespace.key')` in `template()` and any targeted DOM update
4. Add the English defaults to `app/strings.js` (not in the `_lib/` component)

`_lib/` component:

```js
import { t } from '../../strings.js';

// In template():
// t('myComponent.title'), t('myComponent.confirm'), etc.
```

`app/strings.js` (first import in `app/main.js`) — English defaults:

```js
import { defineStrings } from '../_lib/core/strings.js';

defineStrings({
  'myComponent.title':   'Title text here',
  'myComponent.confirm': 'Confirm',
  'myComponent.cancel':  'Cancel',
});
```

Additional locale packs imported in `app/main.js` after `app/strings.js`:

```js
import '../app/locales/fr.js'; // calls defineStrings({ ... }, 'fr')
```

`app/locales/fr.js`:

```js
import { defineStrings } from '../_lib/core/strings.js';

defineStrings({
  'myComponent.title':   'Titre ici',
  'myComponent.confirm': 'Confirmer',
  'myComponent.cancel':  'Annuler',
}, 'fr');
```

## Report format

**For file audits:**
- ✅ Pass — no hardcoded strings, `t()` and `defineStrings()` used correctly
- ⚠️ Warning — string is borderline (e.g. a proper noun or brand name) — flag for developer decision
- ❌ Fail — hardcoded user-visible string, missing import, or missing `defineStrings()` — must fix before completion

**For locale audits:**
- List missing keys per locale file
- List extra keys per locale file (may be renamed keys)
- Count total keys in base locale

**Tier 2 notes:**
- 📋 Note — Intl usage outside wrappers, noted for V4 Intl wrapper module

End with: how many Tier 1 failures (blocking), how many warnings, how many Tier 2 notes.
